# /verify — 端到端验证

## 触发条件
代码变更后、部署前、请求验证功能是否正常。

## 执行流程

### 1. 变更分析
- 通过 git diff 识别变更文件
- 分析影响范围（哪些模块/功能受影响）

### 2. 自动化检测
- **语法检查**: lint/typecheck
- **单元测试**: 运行相关测试文件
- **集成测试**: 如果有 Docker Compose，启动并测试
- **构建验证**: 确认项目可构建

### 3. 功能验证
- 如果项目正在运行：调用健康检查端点
- 如果有 Playwright 测试：运行端到端测试

### 4. 回归检查
- 对比变更前后的行为差异
- 重点关注修改的接口/函数

### 5. 输出
```
## 验证报告
- 变更文件: X 个
- 影响模块: ...
- 语法检查: ✅ / ❌
- 测试结果: X passed, Y failed
- 构建状态: ✅ / ❌
- 风险等级: 低/中/高
```
