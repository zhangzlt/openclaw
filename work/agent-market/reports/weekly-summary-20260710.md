# 工作周进展（2026-07-07 ~ 2026-07-10）

## 一句话说清楚我们在做什么

我们要让一个「机器人」每天自动去 Agent Market 平台（agent.digitalchina.com）逛一圈，把上面 41 个智能体挨个测试一遍——看它们能不能正常回答问题、能不能上传文件、能不能交互点击——然后把结果整理成带截图的飞书文档，自动发到群里。这样整个团队不用人工点开每个智能体，一眼就能知道哪个挂了、哪个好着。

这周，这个系统从零搭到了每天自动运转。

## 07/08（周二）：让机器人学会"聊天"

**干了什么**：
- **Playwright 对话测试引擎**：用 Playwright（模拟真实浏览器的工具）自动打开智能体聊天页面，像真人一样在输入框打字、按 Enter，然后等 AI 回复。
- 第一版只测了 5 个智能体。当天就把 21 个对话型智能体全部覆盖了（分两个平台：feishuapp.cn 和 aily.feishu.cn）。
- 飞书扫码登录 + Token 缓存：每次测试都要登录，不能每次都让导师扫码。写了个 Token 缓存机制——登录一次，30 天内直接用缓存的 Token，不用重复扫码。
- 浏览器崩溃自动恢复：测试 21 个智能体时浏览器偶发崩溃，加了自动重启机制，崩溃了就新建一个继续跑。
- **飞书文档投递**：第一版尝试创建飞书文档，发现应用权限不够（缺少 `docx:document`），临时用消息推送了纯文本摘要。

**通俗理解**：
教会了机器人"打开网页 → 点进聊天 → 打字发消息 → 等回复"。就像教一个实习生怎么手动测试产品——只是我们把它自动化了。中间遇到两个坑：登录态老过期（加了缓存解决）、浏览器老崩溃（加了自动重启解决）。

**成果**：21 个对话智能体，18 个通过（85%），3 个异常。

**提交数**：9 commits

## 07/09（周三）：不仅要会聊天，还要会传文件、点按钮

**干了什么**：
- **Playwright → agent-browser 迁移**：Playwright 太重了，切到 agent-browser（Rust 写的 CDP 工具，更轻更快）。
- **非对话智能体专项测试**：Agent Market 上除了聊天机器人，还有 14 个"功能性"智能体——比如合同比对（要上传文件）、内容审核（要点击按钮）、数据查询（要导航多个页面）。它们不能用"打字聊天"的方式测，写了全新的测试路径：文件上传测试、页面交互测试、通用可访问性检测。
- **Dify API 直连测试**（Agent 63）：有一个智能体在网页上根本看不到聊天界面，它的聊天逻辑走的是内部 API。直接跳过浏览器，用 HTTP SSE 流去测试，0.4 秒就能测完（比浏览器快 25 倍）。
- **投递架构重构**：之前的报告是纯文本，截图是本地路径。改成启动一个本地 HTTP 服务暴露截图文件，报告里用 `![](http://127.0.0.1/xxx.png)` 引用，飞书文档会自动下载上传。
- **回复等待逻辑修复**：之前是发完消息后"傻等 10 秒"，有些智能体 5 秒就回完了，有些要 30 秒。改成轮询检测——每 1 秒看一次页面有没有新内容，连续 2 轮没变化就认为回复完毕，最长等 45 秒。更准，也更快。

**通俗理解**：
第二天，机器人升级了——原来只会跟人聊天，现在会传文件、点按钮、导航多页面。测试范围从 22 个智能体扩展到 36 个。回复等待也从"死等"变成了"智能等"，回应快的智能体不用浪费时间。

**成果**：非对话 14 个智能体覆盖，Dify API 0.4s 极速测试。

**提交数**：9 commits

## 07/10（周四）：最后的冲刺——截图、修 Bug、推到群

这一天最忙，改了 17 次代码。

### 上午：截图位置大作战

**问题**：飞书文档里 30 张截图全部堆在文档末尾，没有插到对应智能体的「截图：」和「用时：」之间。

**根因**：
1. 用 `feishu_doc write` 写 Markdown 时，`![](localhost_url)` 飞书 API 根本不处理这种本地链接
2. 用 `upload_image + block_id` 时，`block_id` 被当成"父级区块"而不是"插到它后面"

**解决方案**：发现一个关键规律——`upload_image` **不传 block_id** 时会自动追加到文档当前末尾。所以改成交替模式：
```
append("智能体A文字...截图：")  →  文档末尾新增文字
upload_image("A的截图.png")    →  截图紧随其后
append("用时：3s")              →  用时跟在截图后面
```
一个智能体，三步循环。30 张截图全部精准插入 ✅。

### 下午：灾难级的三连 Bug

巡检脚本跑完，发现 **22 个对话智能体全部失败**，全是同一行报错：

```
NameError: name 'screenshot_dir' is not defined
NameError: name 'response_text' is not defined  
NameError: name 'aid' is not defined
```

**原因**：对话和非对话测试走了两套代码路径。重构时 copy-paste 了非对话路径的变量名到对话路径，但两边变量名不一样（一个是 `agent_id`，一个是 `aid`）。三个变量名拼写错误连锁炸穿。

修复：逐个修改变量名，提交 → 重跑 → 确认通过。三连修复后通过率从 6/36 跳到 **28/36（78%）**。

### 下午到傍晚：打磨细节

- **5 个特殊智能体交互适配**：有些智能体的页面结构比较特殊（加载慢、需要多步操作），写了自定义交互步骤（type → click → scroll 等），验证通过
- **App not found 检测**：4 个智能体点进去显示"应用不可访问"/"App not found"，原先会报上传失败。加了页面内容检测，遇到这种直接标记 `unreachable`，不再误报
- **严重度排序**：报告里 errors → ok → skipped，一眼看到问题
- **群聊推送打通**：之前 bot 不在 Agent Market 群里，拉进去后推送成功 ✅
- **判定标准放宽**：之前 LLM 会评估回复"质量不及格"就判失败。导师说只要返回了回复就算有效。去掉这段代码，等待时间也拉到 60 秒给慢智能体够时间渲染
- **SKILL.md 重写**（v1.1.0 → v2.0.0）：整个操作手册翻新，记录下所有踩过的坑和经验
